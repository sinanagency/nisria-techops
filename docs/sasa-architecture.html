1. What this is, in plain terms

Sasa is an office, not a single brain. A message arrives like a customer at a front desk. A receptionist routes it to the right clerk. Each clerk has only their own equipment. The clerk's job is to understand what is wanted; the system does the action and writes the reply. That separation is the whole design.

The bot has moved through three versions. Mono: one brain with every tool, no routing, easily confused. Mesh (live now): a router plus seven scoped specialists on one shared engine, with an honesty chain that refuses false claims. Rails (in progress): narrow the model to understanding only, and make the action and the reply deterministic, so a clear command produces one correct, gradable result.

The recurring failure was never routing or tools. It was the last step: letting the model write the "what happened" sentence freely, where it could say "Logged" for a payment that was only staged. The fix is to take that pen away. This document maps every node so each one can be improved on purpose, not by guesswork.

Current state (honest)

  • New architecture is deployed to prod and verified live (the new /api/eval/replay route answers on the apex; a real signed webhook drove the full pipeline).
  • The bot is gated by maintenance right now: only the owner number is served; members get a single notice. Lifting it is one env flip plus redeploy.
  • The brain runs on a funded Anthropic key set in prod. Swap to a dedicated bot key when convenient.

3. The runtime logic tree, node by node

Each node lists what it does, where it lives, its current weak point, and the concrete improvement. This is the work list.

1Webhook (fast intake)app/api/whatsapp/webhook/route.ts:61
Verifies Meta's HMAC signature, resolves the sender to a contact, stores the inbound message, enqueues a job, triggers the worker, and returns 200 within Meta's timeout. The slow brain runs separately so Meta never sees a timeout.
Weak point: a bad or missing WHATSAPP_APP_SECRET rejects everything; media arrives as a reference that must be fetched later.
Improve: add a dead-letter log for signature failures and malformed payloads so silent drops are visible, not invisible.
2Gate 1 — is the sender allowed?worker/route.ts:104
Allowed only if role is admin (owner/founder on the operators allowlist) or a team member with bot_access=true. Everyone else is logged as whatsapp.ignored and gets no reply. The 727 line is invite-only.
Weak point: a legitimate new team member with no bot_access flag is silently ignored, which looks like the bot is broken.
Improve: a one-line "you are not enrolled yet, ask Nur" reply for known-but-unflagged team numbers, so silence never reads as failure.
3Gate 2 — maintenanceworker/route.ts:213
When MAINTENANCE_MODE=1, only the allowlisted owner number gets full service. Everyone else gets one canned maintenance notice (throttled to once per six hours) and nothing else runs: no brain, no tools, no writes. This gate is currently ON.
Weak point: the gate is all-or-nothing and lives in an env var, so toggling needs a redeploy; there is no partial / read-only mode.
Improve: a soft mode that lets reads through but blocks writes during a quality pass; move the toggle to a DB flag so it flips without a redeploy.
4Coalesce the burstworker/route.ts:396
A durable per-contact claim merges several quick messages ("salary 15k", "+5k transport", "for Mark") into one turn so the bot answers the whole thought, not each fragment. Losers defer to the winner.
Weak point: the coalesce window is a fixed wait; too short splits a thought, too long feels slow.
Improve: end the window early on a sentence-final signal (a question mark, "thanks", a long pause) instead of always waiting the full timer.
5Window-open flushworker/route.ts:431
When a contact messages in, any reply that was held earlier (because WhatsApp's 24-hour window was closed) can now be delivered. Gated by isLive so test and maintenance traffic never trigger a real send.
Improve: surface held-message count to the operator so nothing waits silently for days.
6Orchestrator — route & (maybe) decomposelib/agents/orchestrator.ts:69
Media goes through an intake pipeline to a domain. Text goes to routeMessage (regex first, Haiku model as fallback), returning a domain and a confidence. Low-confidence or multi-intent messages are split into up to three steps, each run as its own specialist. There is no monolith fallback by design.
Weak point: routing is score-by-keyword-length; "remind me to send X" can tip to comms instead of work. Routing was ~86% on the hard set.
Improve (the big lever): this is where the Rails plan lands. Replace free tool-choice downstream with a structured parse (intent + slots), so the path becomes route to deterministic action. See section 7. Emits mesh.routed for measurement.
7Specialist — scope the tools, focus the lanelib/agents/specialists/index.ts:54
Looks up the domain's tool list, fails closed if the scope is somehow empty (never runs unscoped), injects the domain focus plus the no-scope-leak rule, and calls the shared engine. This is the security wall: a money turn physically cannot reach a people tool.
Weak point: the focus is prose the model can still drift around; tool choice still happens inside the engine, not here.
Improve: move tool choice out of the engine and into a deterministic intent to tool table here, per the Rails plan.
8Shared engine — the tool looplib/agents/sasa.ts (runSasa)
Builds the system prompt (persona, Brain grounding, the real current date via the clock), then loops: the model proposes a tool, a dispatch-time scope check rejects any tool outside this domain, the tool runs, the result is fed back. Staging tools (payment, merge, delete) return "Ready to ... reply yes" with a staged flag.
Weak point: the model can still call the wrong in-scope tool, or loop asking instead of acting.
Improve: with the parse layer in place, the engine's tool loop shrinks to "execute the parsed action," removing most wrong-tool turns.
9Finalize — the honesty chainlib/agents/sasa.ts (finalize)
The reply passes through ordered guards. PRIMARY (new): deterministic staged-confirmation — if any tool staged something, the reply IS the tool's own line, so the model can never say "Logged" for a staged payment. Then: send-state truth, fake-staging, completed-but-staged, plural-send, fabricated-number, the meta/scope-leak strip, apology cap, and a completion-mismatch backstop. Finally WhatsApp formatting and chunking.
Weak point: the later guards are catch-after-the-fact; each new lie shape needs a new guard (the game we are leaving).
Improve: as Rails extends, every action's confirmation comes from the tool result, so the guards become a near-silent backstop. A guard firing should be an alert, not normal.
10Orchestrator finalize-with-guardlib/agents/orchestrator.ts:149
A last cross-domain leakage check on the real tools that ran, then emits mesh.completed with the tools and step count for measurement.
Improve: include the parsed intent and confidence in the telemetry so every turn is gradable against the golden set automatically.
11Sendlib/whatsapp.ts (sendTextAndLog)
Sends the reply over WhatsApp and persists it, except in dev mode (owner/harness), which reroutes and skips persistence with a [DEV] prefix.
Improve: confirm delivery via Meta's status callback and retry or flag on failure, so a dropped send is never assumed delivered.

5. The honesty chain in detail (finalize)

Guards run in order. The first that matches rewrites the reply and the rest stand down. The top one is the new deterministic path; the others are the historical catch-after guards that Rails will make near-silent.

GuardWhat it catches
deterministicStagedConfirm newAny staged action (payment, merge, delete). The reply becomes the tool's own honest "Ready to ... reply yes" line, so the model cannot narrate a staged action as done.
send-state truth"Did you message X?" answered from the real outbound log, not the model's guess.
fake-staging"Reply yes to confirm" text when no staging tool actually ran.
completed-but-staged newBackstop: "Logged" when only a staging tool ran. Now superseded by the primary path, kept as a safety net.
plural-send mismatch"Both messages sent" when only one actually went.
fabricated-numberA KES/USD figure not in the user's words or any tool result.
meta/scope-leak strip new"I'm scoped to comms tools only", "the rules I run on" and similar internal leaks.
apology capRepeated "I'm so sorry, Nur" condolence openers on routine turns.
completion-mismatch backstopAny "done/created/sent" claim with no category-matched tool success this turn.

7. How we improve it (the Rails plan)

One change, applied per domain: narrow the model to understanding, make the action and the reply deterministic.

  TODAY (mesh)                         TARGET (rails)
  route -> specialist:                 route -> specialist:
    LLM picks tool + writes reply         1. LLM PARSE -> {intent, slots, confidence}
    (wrong-tool + wrong-words bugs)       2. deterministic intent -> tool (scoped)
                                          3. execute, capture result
                                          4. templated confirmation from the result
                                          5. fallback to free chat only when
                                             intent = conversation / low confidence

Why this reaches near-100% on clear commands

  • The model does one job it is good at: turn messy, multilingual phrasing into a structured intent and slots.
  • Tool choice becomes a lookup table over the specialist's already-scoped tools, so wrong-tool turns disappear.
  • The confirmation is rendered from what the tool actually did, so the bot cannot lie about an outcome.
  • Every recognized action has one correct result, so the bot becomes perfectly gradable against a golden set.

Sequence

  • Done: staged-action confirmations (money, merge, delete) deterministic; leak and routing fixes.
  • Next: build the parse layer for work first (largest cluster), then money, people, comms, knowledge, programs.
  • Each domain: intent catalog, intent to tool table, slot validators, confirmation templates, walls.
  • Measure once at the end against the golden set on a safe sandbox deploy (sends dead, never prod), to respect key spend.

References in the repo: spec 005 (specs/005-deterministic-rails), ADR 0015 (docs/decisions/0015-intent-parse-then-execute.md), knowledge-tree nodes 406 to 408.

8. How to operate it

  • Unlock: set MAINTENANCE_MODE=0 in prod env, redeploy. Lock: set it to 1, redeploy. The allowlist (owner number) always gets through.
  • Brain key: ANTHROPIC_API_KEY in prod (type sensitive). It must be a funded key or the bot goes silent. Verify with a live brain test before unlocking.
  • Deploy oracle: never trust the push. Confirm command.nisria.co is aliased to the new deployment and a discriminator route answers, then verify the brain with one real signed message.
  • Tests: npm run walls (80 walls) plus tsc must be green; predeploy.sh gates the prod push.
