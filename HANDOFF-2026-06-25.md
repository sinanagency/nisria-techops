## The Bug

**`router.classified` events never appear in the `events` table, despite confirmed execution of the orchestrator and router.**

Evidence from the events table:
- `mesh.debug_entry` appears → orchestrator called
- `mesh.debug_router_entry` appears → `routeMessage` entered
- But NO `router.classified` event EVER (0 results across all queries)

Latest test: TEST_MESH_011 at ~14:28 UTC showed `whatsapp.message_in` → `task.assigned` → `task.alert_sent` → `whatsapp.message_out` but no `router.classified`.

The monolith IS running (falls back via `runSpecialist → runSasa` or direct `runSasa`). The `sasa.honesty_guard_substituted` event sometimes appears. Normal replies are sent — the bot WORKS but we can't verify mesh isolation.

### Key debugging attempts (all in working tree / uncommitted)
1. Dynamic import of `emit` from events module → no events
2. Static import of `emit` from events module → no events
3. Direct Supabase insert via `admin().from("events").insert(...)` bypassing `emit` entirely → no events
4. File write to `/tmp/router-debug.log` via `require("fs")` added but not tested

### Current state of uncommitted changes

**router.ts** (at `/Users/milaaj/Code/nisria-sr/platform/lib/agents/router.ts`):
- Has static import `import { admin } from "../supabase-admin"` (was changed from `import { emit } from "../events"`)
- `emitRouterTelemetry` now writes to `/tmp/router-debug.log` via `require("fs")` AND calls `admin().from("events").insert(...)` directly
- `emitRouterTelemetry` is AWAITED at every call site (so if it throws, routeMessage will fail)

**events.ts** — reverted to original (no modifications)

### Why router.classified never appears (theories)

1. **Router code not reaching production** — despite `vercel deploy --prod --yes` completing, maybe the build is cached or using a different source. We could NOT verify the git commit hash of the running deployment (Vercel CLI and API both have issues with this project).

2. **`emitRouterTelemetry` silently throws** — The function now has NO try/catch (I removed it). If `admin()` fails, it throws all the way up, triggering the worker's stuck message handler. But we DON'T see stuck messages → so it doesn't throw.

3. **Supabase insert silently fails** — Even though `admin()` returns a client, the `.insert()` call might return an error (not throw). The `admin().from("events").insert(...)` call returns PostgREST response `{ error, data }`. If `error` is set and not checked, the operation silently failed.

4. **Build caching issue** — Vercel might be using a cached build that predates the router telemetry code. (See issue with `vercel inspect` not showing git commit.)
