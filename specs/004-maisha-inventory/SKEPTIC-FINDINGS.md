# Maisha Inventory — Adversarial Skeptic Pass (assume-connected)

> 6 adversarial agents, each tasked to REFUTE production-readiness against the LIVE repo. Verdict: **unanimous NOT-CONNECT-READY.** The sandbox proves the happy-path LOGIC; it is not a migration and several logic bugs + live bugs surfaced. Findings below are deduped and prioritized. All carry file:line in the source agent reports.

## Verdict
The 20/20 green suite is green because PGlite uses `text` ids, no RLS, no FKs to `team_members`, empty tables, and a regex stand-in for the LLM — every green test sidesteps the exact live constraint that will reject the real write. **Green sandbox ≠ connect-ready.** Three buckets below: (A) sandbox logic bugs fixable now, (B) migration/integration work for connection, (C) pre-existing LIVE bugs the exercise surfaced (independent value).

---

## A. Sandbox logic bugs — fix in the sandbox now (cheap, high-value)

1. **`recordSale` is non-atomic and lies on partial failure (BLOCKER).** Selling a `delivered`/`returned` item: the refused `transitionState` return is ignored, yet it returns `{ok:true,"recorded sale"}` + spawns a ship task. Revenue row with no state change, false "done." `tools.ts:160-168`. → honor the transition result; wrap sale+transition+task in one transaction.
2. **`batch_tag` per-item constant swallows legit resale revenue (BLOCKER).** Key is `inv:<tracking>:sale`; after return→restock→resell, the second real sale returns `deduped:true` and the money vanishes. `tools.ts:155-157`. → key per sale-event, not per item.
3. **`consumeMaterials` has no stock floor — quantity goes negative (HIGH).** `quantity = quantity - $1`, no `GREATEST(...,0)`, no `CHECK(quantity>=0)`. `tools.ts:143`, `schema.sql:68`.
4. **6 registered tools have NO handler (HIGH).** `correct_record`, `upsert_end_product/supply/textile`, `get_lifecycle`, `record_payment_link` are in `guard.ts` `INVENTORY_TOOLS` but unimplemented in `tools.ts`. The guard meta-check verifies a *flag*, not a *handler* — its headline claim is partly hollow. `correct_record` missing = operators have no amend path → duplicate/contradictory rows. → implement or de-register; extend `verifyGuardRegistration` to assert a handler exists.
5. **Orphan drafts grow unbounded; the sweep is fiction (HIGH).** Every image makes a `draft`/`pending` row; `pg_cron` sweep is a comment only. In a real group every meme/receipt becomes a permanent fake `end_product` draft, hidden by the `enriched=TRUE` filter. `tools.ts:50-60`, `schema.sql:128`. → build the sweep + nudge, or don't pre-create the inventory row until classified.
6. **Pre-classification typed as `end_product` before classify (HIGH).** Orphans keep a false discriminator. `tools.ts:52`. → type as `unknown`/null until enriched.
7. **No transactions on any multi-write tool (MED).** `persistPendingImage`, `enrichRecord`, `transitionState`, `recordShipment`, `consumeMaterials` all autocommit per-await; a mid-tool throw leaves partial state (e.g. inventory row with no `pending_enrichment` tracker = unreachable orphan). → RPC/transaction wrap.
8. **Failure-path tests entirely missing.** No supply/textile enrichment test (every test feeds a tracking#→always end_product), no returns→restock→resell, no orphan, no negative stock, no `correct_record`, no sale-on-non-sellable-state, no currency-mismatch sale. The suite tests success, never failure.

## B. Migration & integration work — required before connection

9. **There is no migration. (BLOCKER)** `schema.sql` is greenfield `CREATE` with `id TEXT` vs live `uuid`; it writes `messages`/`tasks`/`payments` as CREATE (→ `relation already exists`), not ALTER. `item_type NOT NULL` with no default annihilates every existing `inventory` row (23502); no backfill decided for legacy rows; ADDs lack `IF NOT EXISTS` (`reply_to_external_id` already live → 42701); new `UNIQUE`s can fail on dupes; 4 new tables miss the live RLS-enable sweep. → write a real `db/migrations/*.sql` against live (uuid/RLS/FK/non-empty), backfill `item_type` first.
10. **The ingest/binding pipeline isn't wired to prod at all (BLOCKER).** `pending_enrichment`, `persistPendingImage`, `binder`, `asset_id` on the group path: zero presence in `platform/`. **Mode 1 (reply-bind) is structurally dead** until the Baileys userbot ships the quoted message's wamid (live only gets `quoted_text` string). Group path uses raw `storage.upload`+`media_path`, never `storeMedia`/`asset_id`. → wire persist + `storeMedia`+`asset_id` + persist `reply_to_external_id` + move the SYSTEM gate to ingest.
11. **Group path has no job queue → inline 300s timeout on photo bursts (HIGH).** `runSasa` runs synchronously in the POST handler; a 20-photo catalog drop with per-message `storeMedia`+vision piles up. The 1:1 path deliberately moved the brain to a drained queue + per-sender coalescing + durable `awaiting_note` slots — the group path threw all three away. The "ask once" ambiguity has no durable slot, so a clarification re-enters as a fresh follow-up and binds to the newest photo. `ingest`/`worker` routes.
12. **Loose-follow-up binder is newest-wins → mis-binds in busy groups (HIGH).** Caption about an *older* photo lands on the newest; a catalog drop makes every caption "ambiguous"; the phone-wall is the only thing preventing cross-maker bleed and it's load-bearing on userbot sender accuracy. → ordinal/adjacency binding, not newest-wins.
13. **sasa.ts has 13 disjoint registration sites, no live meta-check (BLOCKER-class).** `COMPLETION_TOOLS`, a NEW `SHAPE_INVENTORY` + shape-array entry + `INVENTORY_TOOLS` requiredTools, `AGENT_COMPLETION` verb list (missing shipped/delivered/sold/moved → fabricated "I shipped it" passes unchecked), `TEAM_TOOL_NAMES`, `READ_TOOLS`, `STAGING_TOOLS`, `SEND_TOOLS`, `stubTool` switch, the worker confirm-`kind` switch, `carriesMoney`/`FINANCE_GROUNDING` vocab. SHAPE words collide with `SHAPE_MONEY` (a true sale with a price → checked against PAYMENT_TOOLS → hedged). → port `verifyGuardRegistration` as a real unit test over the LIVE Sets; add the inventory shape + verbs.
14. **Finance: `logExpense` INSERT hard-fails (BLOCKER).** Live `payments` has no `source` or `batch_tag` column; the INSERT and the idempotency SELECT both throw on connection. → migration adds both.
15. **Finance: `inventory_sales` is read by ZERO live surfaces (BLOCKER).** Revenue is captured in a silo; Treasury/finance read only donations/payments/bank_transactions. Reconciliation claim is hollow. → a live surface must read it.
16. **Finance: cost pollutes donor-facing NGO totals (HIGH).** `loadExpenses`/Treasury sum `payments` with no `source` filter (they don't even select `source`). Maisha cost leaks INTO the nonprofit operating-spend headline while revenue stays invisible — the worst asymmetry. → `.neq("source","maisha_inventory")` everywhere + select the column.
17. **Finance: AED reopens the $14.85M blend class (HIGH).** Live finance buckets only USD/KES; an AED row is either silently dropped from headlines or, if currency is null, coerced `|| "KES"` and summed at face value. Live `sumByCurrency` accepts anything (sandbox throws). No `org_profile.fx_rates`; Treasury hardcodes `129` KES/USD with no AED leg. → add AED to the live currency model or refuse; implement stamped fx_rates.

## C. Pre-existing LIVE bugs surfaced (independent value — worth fixing regardless)

18. **Worker confirm-switch silently commits unknown kinds (BLOCKER, live today).** `worker/route.ts:485-513`: the `else` catch-all marks a `pending_action` `committed` and replies "Done. Logged …" for any `kind` it doesn't know — so a staged `record_sale` "yes" would report success with zero write. Honesty inversion already latent. → fail-closed on unknown kind.
19. **`learnMemberPhone` identity hijack (HIGH, live today).** A group member who sets their display name to a phone-less roster member's first name gets their phone auto-bound to that member → inherits `team` tier (+ bot_access). No confirmation. `group/ingest/route.ts:103-114`. → admin-confirmed enrollment.
20. **No per-actor authorization on inventory writes (HIGH).** Authority is tier-only; any team phone can mutate/sell/ship/consume any record; `created_by` is the forgeable `senderName`; live `add_inventory_item` hardcodes the actor to "Nur". → per-actor `created_by` checks.
21. **Customer path is both dead and a loaded gun (HIGH).** `operatorOf` drops unknown numbers, so a customer texting a token is ignored (feature non-functional); and the token is an unbound, guessable, non-expiring, unrate-limited bearer secret (`TOK-<firstname>1`, length≥6 the only check) — enumerable the moment a customer surface is wired. → CSPRNG token bound to buyer phone + expiry + rate limit; typed customer-safe DTO (it currently leaks `tracking_no`).
22. **Prompt injection on the extractor (HIGH, on LLM swap).** When the prod LLM replaces the regex, untrusted group text/photo-text drives `transitionState` with no instruction/data separation, no field allowlist, no schema validation. Sasa's honesty guards inspect reply *text*, not tool *inputs*. → schema-validated, instruction-isolated extractor before any tool input.

## Meta-lesson
The sandbox's own safety claim (`verifyGuardRegistration` "converts the silent bite into a caught error") is itself partly hollow — it checks a flag, not a handler, and 6 tools are registered with no implementation. The skeptic pass caught the catcher. The genuinely solid parts: the honesty-rewrite model and the currency-law primitive (sandbox `sumByCurrency` throws on unknown currency — stricter than live, which is the right direction). Everything else needs the work above before "connected" is anything but a liability.
